# ACME protocol + Let's Encrypt — the deep-dive

> **Goal**: by the end you can answer the killer question — **"Walk me through how Let's Encrypt verifies you own a domain before issuing a cert."** — naming the ACME state machine, the two-key model, and the production gotchas around rate limits, staging, and the CA cert chain. You can also explain why 90-day certs are a security feature, not a limitation.

> Start with [acme-letsencrypt-simple.md](./acme-letsencrypt-simple.md) if you haven't read it. The notary-stamp analogy is the spine of everything here.

---

## The senior framing — ACME exists because humans are bad at cert renewal

Before ACME (RFC 8555), getting a TLS cert meant:
1. Generating a key and CSR manually.
2. Submitting it to a CA's web UI.
3. Waiting for human review (days for OV/EV certs).
4. Downloading the cert and manually deploying it.
5. Setting a calendar reminder for renewal.
6. Forgetting the reminder.
7. Outage.

Let's Encrypt's insight (2014–2016): **domain validation is fully automatable.** You don't need a human to verify that `example.com` is controlled by the person requesting a cert — you just need the requestor to demonstrate live control. ACME is the protocol that codifies this.

The senior point: ACME + Let's Encrypt moved TLS from "expensive thing finance approves quarterly" to "infrastructure primitive automated by cert-manager." That shift is why nearly every public K8s cluster now has valid TLS without an operations team managing cert renewals.

## The ACME state machine

ACME is a REST protocol over HTTPS. Every action is an API call with a signed body. Here's the full flow from zero to cert:

### Step 1 — Account creation (`newAccount`)

The first time your ACME client runs, it creates an account with Let's Encrypt. An account is identified by an **account key** — an ECDSA or RSA keypair generated by the client.

```
POST /acme/new-account
{
  "termsOfServiceAgreed": true,
  "contact": ["mailto:admin@example.com"]
}
```

Let's Encrypt stores the account and returns an account URL. All subsequent requests include a `kid` header pointing to this account URL.

**Important**: the account key is not the cert key. It's your identity within the ACME system. cert-manager stores this as a K8s Secret (the `privateKeySecretRef` in your `Issuer`). If you lose the account key, you can create a new account; you just can't revoke old certs via the API without it.

### Step 2 — New order (`newOrder`)

You request a certificate for specific identifiers (domain names):

```
POST /acme/new-order
{
  "identifiers": [
    {"type": "dns", "value": "example.com"},
    {"type": "dns", "value": "www.example.com"}
  ]
}
```

Let's Encrypt responds with an `Order` object that contains:
- A list of authorization URLs (one per identifier)
- A finalize URL (used later to submit the CSR)
- The order's expiry time (you must complete challenges within this window — typically a few days)

### Step 3 — Authorizations and challenges

For each authorization URL, Let's Encrypt returns a list of challenges. You only need to complete **one** challenge per identifier.

Each challenge object looks like:

```json
{
  "type": "http-01",
  "url": "https://acme-v02.api.letsencrypt.org/acme/challenge/...",
  "token": "SxVzN8aKQCqzqoaLMl9a7..."
}
```

The challenge token is a random value from Let's Encrypt. You use it (combined with your account key) to compute the key authorization:

```
keyAuthorization = token + "." + base64url(SHA256(accountKey.publicKey))
```

This is the value you place in the HTTP response or DNS TXT record. Let's Encrypt verifies it against your account key — proving that whoever placed the challenge controls both the domain and the account key.

### Step 4 — Complete the challenge

For HTTP-01: make the key authorization available at:
```
http://<domain>/.well-known/acme-challenge/<token>
```

Then notify Let's Encrypt you're ready:

```
POST /acme/challenge/<id>
{}
```

Let's Encrypt fetches the URL from multiple vantage points (they use multiple POPs to prevent BGP hijacking attacks). If the token matches across a quorum of vantage points, the challenge is "valid."

### Step 5 — Wait for authorization validity

Poll the authorization URL until `status` is `valid`. If any challenge fails, status goes to `invalid` — and the order fails. You'd need to create a new order.

### Step 6 — Finalize the order (submit CSR)

Generate a fresh keypair (the **certificate key** — completely separate from the account key). Create a CSR (Certificate Signing Request) with the domain names. Submit it:

```
POST /acme/order/<id>/finalize
{
  "csr": "<base64url-DER-encoded-CSR>"
}
```

Let's Encrypt validates the CSR, signs the cert, and transitions the order to `valid`.

### Step 7 — Download the certificate

```
POST /acme/order/<id>/certificate
```

Returns a PEM chain: your leaf cert, the intermediate CA cert, and (sometimes) the root. The private key never left your system — only the CSR went to Let's Encrypt.

## Account key vs certificate key — why two?

This confuses people. Here's the precise distinction:

| Key | Generated by | Stored where | Used for | Rotated when |
|---|---|---|---|---|
| **Account key** | ACME client | cert-manager Secret (or Certbot's config dir) | Signing all ACME API requests (authentication) | Rarely — account key compromise means key rollover via ACME API |
| **Certificate key** | ACME client | cert-manager Secret (or wherever the cert lands) | The private key for TLS handshakes | Every cert renewal (cert-manager generates a fresh one by default) |

The account key is like your identity document — stable, rarely changed, proves who you are to Let's Encrypt. The certificate key is the actual TLS credential — ephemeral, rotated frequently, specific to a cert.

In cert-manager, the account key is stored in the `privateKeySecretRef` of the `Issuer`. The certificate key is stored alongside the cert in the `secretName` of the `Certificate`. They're in different Secrets.

**Production gotcha**: if your `Issuer`'s account key Secret is deleted, cert-manager creates a new account. The new account doesn't know about old certs and can't revoke them via API. You lose rate limit allowances accrued under the old account. Always back up the Issuer's account key Secret.

## Rate limits — the ones that will catch you

Let's Encrypt production rate limits (as of 2025):

| Limit | Value | Notes |
|---|---|---|
| Certificates per Registered Domain per week | 50 | "Registered domain" = `example.com`, not subdomains individually |
| Duplicate certificates per week | 5 | Same exact set of SANs, same or different order |
| Failed validations per account per hostname per hour | 5 | Accumulates across all accounts if from same IP |
| New orders per account per 3 hours | 300 | |
| New accounts per IP per 3 hours | 10 | |
| Pending authorizations per account | 300 | |

**The "registered domain" gotcha**: if you have 50 subdomains under `example.com` and they all renew in the same week, you're at the limit. cert-manager staggers renewals based on cert expiry, but a mass deployment (new cluster, CI environment per branch) can hit this fast.

**The duplicate cert limit**: issuing the same cert more than 5 times in a week (same SANs) is blocked. This catches: deleting and recreating a `Certificate` resource repeatedly, debugging a cert-manager config by creating the same cert over and over. The fix: use staging for debugging.

**The failed validation limit**: 5 failures per hostname per hour, per account. If your HTTP-01 endpoint isn't reachable and cert-manager retries, you can exhaust this. cert-manager has exponential backoff; don't restart it manually trying to "force" it.

## Staging — non-negotiable

The staging environment:
```
https://acme-staging-v02.api.letsencrypt.org/directory
```

Staging issues real ACME-valid certs using the "Fake LE Intermediate" CA. Browsers don't trust this CA, but your automation can test against it. Rate limits are 30,000 certs per domain per 72 hours — essentially unlimited for development.

In cert-manager, this means two `ClusterIssuer` resources: one for staging, one for production. Develop against staging. Switch to production when confirmed working.

```yaml
# Staging — for testing
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-staging
spec:
  acme:
    server: https://acme-staging-v02.api.letsencrypt.org/directory
    email: platform@example.com
    privateKeySecretRef:
      name: letsencrypt-staging-account-key
    solvers:
      - http01:
          ingress:
            class: nginx

---
# Production — only after staging works
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: platform@example.com
    privateKeySecretRef:
      name: letsencrypt-prod-account-key
    solvers:
      - http01:
          ingress:
            class: nginx
```

**CI pipeline pattern**: ephemeral CI environments should use staging. Even if the test environment needs a "real-looking" cert, staging + a `--insecure-skip-tls-verify` test flag beats burning production rate limits.

## Let's Encrypt CA cert chain — the story you need to know

Let's Encrypt's chain of trust:

```
ISRG Root X1 (self-signed, RSA 4096)
└── Let's Encrypt R10 (intermediate, ECDSA)    ← certs issued today
└── Let's Encrypt R11 (intermediate, ECDSA)    ← certs issued today
```

They also maintain cross-signed intermediates via IdenTrust (DST Root CA X3 historically), but that chain expired in September 2021. **The 2021 Android incident**: devices running Android 7.1.1 and older had DST Root CA X3 in their trust store as the cross-sign anchor, but ISRG Root X1 was not in their trust store. When the cross-sign expired, those old Android devices could no longer validate Let's Encrypt certs.

Let's Encrypt offered an "alternate chain" workaround (a special cross-sign via ISRG Root X1 signed by IdenTrust for Android compatibility), but the real lesson: **your cert chain depends on your CA's trust store decisions.**

For Kubernetes workloads: Go applications bundle their own trust store (the `crypto/x509` pool, populated from the system CA bundle in the container). If your base image has an outdated CA bundle (`ca-certificates` package not updated), it may not trust ISRG Root X1 and Let's Encrypt certs will fail validation.

**Production fix**: always install and update `ca-certificates` in your Docker images. In distroless images, the CA bundle is embedded — use recent distroless images.

## Why 90-day certs are a security feature

Let's Encrypt defaults to 90-day certs. This is intentional:

1. **Forces automation**: a cert that expires in 90 days gets renewed at day 60 by automation. An annual cert gets renewed "when someone remembers." Annual certs accumulate unrotated private keys for years.
2. **Limits compromise window**: if a private key leaks, the exposure window is bounded. A 90-day cert leaking has a max exposure of 90 days before the key is invalid (and typically ~30 days if the automation already requested a renewal).
3. **Keeps the CA's records current**: Let's Encrypt can verify domain ownership frequently, ensuring the cert holder still controls the domain.

The common mistake: "I'll request a 1-year cert to avoid renewal hassle." This is the wrong optimization. Build the renewal automation; the 90-day discipline is what makes your cert hygiene strong.

## The interview answer in 60 seconds

> "The ACME protocol has roughly 7 steps: account creation, new order (listing the domains), authorization (Let's Encrypt issues a challenge per domain), challenge completion (you prove domain control by placing a token at an HTTP URL or DNS TXT record), authorization polling (wait for 'valid' status), order finalization (submit the CSR — the cert's keypair is generated locally and the public key goes in the CSR), and cert download.
>
> The two-key distinction is important: the account key identifies your ACME account and signs every API call — it's your identity. The certificate key is generated fresh for each cert and is the actual TLS credential. The cert's private key never leaves your system.
>
> Rate limits: 50 certs per registered domain per week in production; 5 duplicate certs per week. Always use the staging environment for testing — it has essentially unlimited rate limits and issues real ACME certs (just from an untrusted CA).
>
> The senior nuance: 90-day certs are a design choice to force automation and limit compromise windows. The 2021 Android chain expiry is a good example of why you should understand the CA trust chain your clients use — and keep your base image CA bundles up to date."

## Self-test drills

### 1. Walk me through how Let's Encrypt verifies you own a domain before issuing a cert.

**Reference answer:**
- You submit a `newOrder` for `example.com`.
- Let's Encrypt returns an authorization with a challenge (HTTP-01 or DNS-01).
- For HTTP-01: you compute `keyAuthorization = token + "." + base64url(SHA256(accountPublicKey))` and serve it at `http://example.com/.well-known/acme-challenge/<token>`.
- You notify Let's Encrypt you're ready; it fetches the URL from multiple vantage points.
- If the key authorization matches, the challenge is valid. The order can then be finalized.
- For DNS-01: same key authorization, but placed as a DNS TXT record on `_acme-challenge.example.com`.
- Only someone who controls the domain can place either of those — that's the proof.

### 2. You're debugging a cert-manager setup and hitting rate limit errors. What do you do?

**Reference answer:**
- Switch to the Let's Encrypt staging environment immediately — it has relaxed rate limits.
- Check which limit you hit: `Certificates per Registered Domain` (50/week) vs `Duplicate Certificates` (5/week) vs `Failed Validations` (5/hour/hostname).
- If duplicate cert limit: you've been creating/deleting the same `Certificate` resource. Use staging for config testing.
- If failed validation limit: cert-manager is retrying a broken challenge setup. Fix the underlying issue (check `Challenge` events), don't just restart cert-manager.
- If account cert limit: consider requesting a rate limit increase from Let's Encrypt (for production workloads with legitimate need).

### 3. What is the `privateKeySecretRef` in a cert-manager `Issuer`, and what happens if that Secret is deleted?

**Reference answer:**
- It stores the ACME **account key** — not the certificate key. This identifies the ACME account at Let's Encrypt.
- If deleted, cert-manager creates a new account on next use. The new account has fresh rate limit counters (potentially lower headroom). Old certs can't be revoked via the API. Any certificates issued under the old account continue working until expiry — the deletion doesn't affect existing certs.
- Best practice: treat this Secret as load-bearing. Back it up separately. Use Velero or ETCD backups that cover it.

### 4. Why can't HTTP-01 issue wildcard certs?

**Reference answer:**
- HTTP-01 proves control of a specific hostname (`www.example.com`). Wildcard certs (`*.example.com`) represent every possible subdomain.
- Let's Encrypt can't verify every possible subdomain via HTTP — there's no single URL to check.
- DNS-01 proves control of the DNS zone for `example.com`, which by definition covers all subdomains.
- ACME spec (RFC 8555) explicitly requires DNS-01 for wildcard identifiers.

## The 4 dimensions (senior framing)

- **Tech**: the ACME state machine (newAccount → newOrder → authorization → challenge → finalize → cert download); account key vs cert key distinction; HTTP-01 needs port 80 public; DNS-01 needs DNS API credentials; rate limits are per registered domain not per subdomain; 90-day certs forced by design.
- **People**: teams often don't know about the staging environment and burn production rate limits during debugging. Document the two-issuer pattern (staging + prod) in your runbook. Make staging the default for any CI/test environment. New engineers need to understand that `privateKeySecretRef` is the account key, not the cert — it's confusingly named.
- **CI/CD**: ephemeral CI environments must use staging issuers. Include the `ClusterIssuer` manifest in your Helm chart or Terraform — don't manually create it. In GitOps: the account key Secret should be provisioned by External Secrets Operator (from Vault or AWS Secrets Manager) so it survives cluster recreation.
- **Operations**: monitor cert-manager with Prometheus metrics and the blackbox exporter (see [cert-rotation.md](./cert-rotation.md)). Alert on `Challenge` objects that are stuck (not transitioning to `valid` within 10 minutes). Alert on accounts approaching rate limits — cert-manager doesn't surface this natively; use a dashboard that counts `Order` objects per week. Runbook for "rate limit hit": switch temporarily to staging, identify the root cause, request rate limit increase from Let's Encrypt if legitimate.
